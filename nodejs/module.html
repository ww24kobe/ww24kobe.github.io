# 模块化CommonJS，AMD，CMD规范

# 前言

随着前端js代码复杂度和代码量的提高，JS中的模块化这个概念逐渐被提出来。

# 模块化的好处

  • 提高代码复用率。
  • 方便进行代码的管理。通常一个文件就是一个模块，如.js、.vue模块。

模块有自己的作用域，只向外暴露特定的变量或函数或对象。

目前流行的js模块化规范有以下几个：

  • CommonJS （代表作有nodejs）
  • AMD (代表作有requireJs)
  • CMD (代表作有seaJs)

下面就来介绍他们以及他们之间的区别。

# CommonJs规范

Node.js使用的就是commonJS规范。多通过module.exports导出模块，再用require来加载模块。

如定义一个数学相加的模块:

//main.js
function add(n1,n2){
    return n1+n2;
}

module.exports = add;

使用上面的模块

var add = require('./math.js');
console.log( add(10,20) );

CommonJS采用require方法同步加载模块。在服务端，由于模块文件都存在本地磁盘，读取非常快，所以这样做不会有问题。但是在浏览器端，由于网络IO的原因，更合理的方案是采用异步加载。这就是前端浏览器不支持CommonJs模块的原因。

# AMD规范(Asynchronous Module Definition)

AMD是RequireJS在推广出来的时候制定的规范。

AMD规范采用异步方式加载模块，模块的加载不影响它后面语句的运行。所有依赖这个模块的语句，都定义在一个回调函数中，等到加载完成之后，这个回调函数才会运行。

AMD 模块依赖声明方式：

require(['a', 'b'], function (a, b) {
    // to to ...
})

AMD推崇的是依赖前置。可以很直观的看到依赖哪些模块。

# CMD规范(Common Module Definition)

CMD是SeaJS在推广出来的时候制定的规范。

CMD 模块依赖声明方式：

define(function (require) {
    var a = require('./a');
    var b = require('./b');
    // more code ..
})

CMD推崇的是就近声明，通过内部require方法进行声明。

# ES6---export/import

在ES6模块化规范诞生之前，Javascript社区以及尝试并提出了AMD、CMA、CommonJS等模块化规范。

但是，这些社区提出的模块化标准，还是有一定的差异性与局限性，并不是浏览器和服务器通用的模块化标准，例如：

  • AMD和CMD适用于浏览器端的Javascript模块
  • CommonJS适用于服务端的Javascript模块

因此，ES6语法规范中，在语言层面上定义了ES6模块化规范，是浏览器端和服务器端通用的模块化开发规范。

在ES6中，每一个js文件都是一个独立的模块，我们可以使用 import 关键字引入模块，通过 export 关键字导出模块，也是我们所推崇的，但是由于ES6目前无法在浏览器中完全兼容，当然有些浏览器如chrome支持。在静态页面html中设置如下即可

<sript type='module' src="./base.js"></script>

或者也可以在Node中通过相应插件如babel进行转换在使用。

# 小结

  • AMD：异步加载模块。代表作有requireJS
  • CMD：异步加载模块。代表作有SeaJs
  • 上面两者不同点在于：AMD 推崇依赖前置，CMD推崇依赖就近。
  • CommonJS:多通过module.exports导出模块，再用require来加载模块。只适用于服务端，代表作有nodejs
  • AMD、CMD用于浏览器端（异步），commonjs用于服务端（同步）。
  • 在ES6中可以通过export导出模块，import导入模块，但浏览器尚未全部支持，需要借助babel等插件。

从编码的习惯，AMD 更加简单且严谨，适用性更广，而在 RequireJS 强力的推动下，在国外几乎成了事实上的异步模块标准，各大类库也相继支持 AMD 规范（如jQuery）。
